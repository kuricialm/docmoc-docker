

# Fix: JSON Parse Errors and Upload Auth Issues

## Root Cause

The "JSON.parse: unexpected character" error happens because in Lovable's preview, there's no Express backend running. The Vite proxy to `localhost:3001` fails, returning HTML error pages instead of JSON. One function (`getSharedDocument`) still uses raw `res.json()` which crashes on non-JSON responses.

The "unauthenticated user" upload error is the same issue — the backend isn't reachable, so the upload endpoint returns a non-JSON/error response.

**Important limitation**: Cookie-based session auth cannot work cross-origin (Lovable preview talking to your Docker backend) due to third-party cookie restrictions. The app works correctly when deployed in Docker where frontend and backend share the same origin.

## Plan

### 1. Fix `getSharedDocument` raw `res.json()` call
Replace `res.json()` on line 355 of `src/lib/api.ts` with the safe `readJsonBody` helper already in the file. This prevents the JSON.parse crash.

### 2. Simplify the BASE URL logic
Keep `VITE_API_BASE_URL` support (it's useful) but simplify the normalization — the current logic is fine, no changes needed here.

### 3. Improve error messages for unreachable backend
Add a network error catch in `apiFetch` so when `fetch` itself fails (backend unreachable), the error says "Cannot connect to backend" instead of a cryptic parse error.

### 4. Remove unnecessary Supabase dependency
The `@supabase/supabase-js` package was added but this app uses a custom Express backend, not Supabase. The `src/integrations/supabase/client.ts` and `types.ts` files are auto-generated by Lovable Cloud — we'll leave those untouched (they're harmless) but the app doesn't use them.

### Files Changed
- **`src/lib/api.ts`** — Fix `getSharedDocument` to use `readJsonBody`, wrap `fetch` in try/catch for network errors

### What This Does NOT Fix
Testing the full app in Lovable preview still requires a reachable backend. For frontend-only development in Lovable (styling, layout, component work), the app will show a clear "Cannot connect to backend" error instead of crashing. For full testing, deploy to Docker.
